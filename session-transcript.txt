❯ where is the health check API defined in the django application?              

⏺ It's defined across three files in backend/core/:

  backend/core/views.py — the view itself:
  from rest_framework.decorators import api_view
  from rest_framework.response import Response


  @api_view(["GET"])
  def health_check(request):

  backend/core/urls.py — maps it to health/:
  from django.urls import path
  from . import views

  urlpatterns = [
      path("health/", views.health_check, name="health-check"),
  ]

  backend/config/urls.py — mounts the core app's URLs under api/:
  urlpatterns = [
      path("admin/", admin.site.urls),
      path("api/", include("core.urls")),
  ]

  So the full path is GET /api/health/, which resolves to
  core.views.health_check — that's the one App.tsx in the frontend calls on load
  to show "Backend API status: ok".

❯ what URL do I use to access the health check page?                            

⏺ Depends on where you're accessing it from:

  - From your host machine (browser, curl, Postman):
  http://localhost:8000/api/health/
  - From inside another container on the same docker-compose network (e.g. if
  frontend code called it server-side): http://backend:8000/api/health/

  Note it's a JSON API endpoint, not a page — hitting it returns
  {"status":"ok"}, not HTML. If you want to see it exercised visually, just load
  the frontend at http://localhost:5173 — App.tsx calls that health endpoint on
  mount and renders the result as "Backend API status: ok".

  Also note the trailing slash matters — Django's urls.py defines it as health/,
  so http://localhost:8000/api/health (no trailing slash) will redirect or 404
  depending on APPEND_SLASH behavior; use the trailing slash to be safe.

❯ where are the GET, PATCH and DELETE URLs defined?                             

⏺ They're generated by DRF's DefaultRouter, not hand-written as separate
  GET/PATCH/DELETE lines. Here's the chain:

  backend/jobs/urls.py — registers the viewset, which auto-generates all the
  routes:
  from rest_framework.routers import DefaultRouter
  from .views import JobViewSet

  router = DefaultRouter()
  router.register("jobs", JobViewSet, basename="job")

  urlpatterns = router.urls

  backend/config/urls.py — mounts it under /api/:
  path("api/", include("jobs.urls")),

  The router expands that single register() call into:
  - GET/POST /api/jobs/ → list / create
  - GET/PATCH/DELETE /api/jobs/<pk>/ → retrieve / partial_update / destroy

  backend/jobs/views.py — JobViewSet is where the actual method behavior lives
  per HTTP verb:
  - http_method_names = ["get", "post", "patch", "delete", "head", "options"] —
  explicitly excludes put, since jobs are never fully replaced, only appended to
  via status history
  - list/retrieve/destroy — DRF's default ModelViewSet behavior (destroy
  cascades to JobStatus via the FK)
  - create is default too, but perform_create is overridden to also create the
  initial PENDING status
  - partial_update is fully overridden — it doesn't touch Job fields at all, it
  just appends a new JobStatus row

❯ so are we hard coding the page size on the server size while the page size on 
  the client is configurable?                                                   

⏺ No — the server isn't hardcoded, it just defines a default and a ceiling; the
  actual page size per request is driven by the client. Here's the mechanism, in
  backend/jobs/pagination.py:

  class JobPagination(PageNumberPagination):
      page_size = 10              # used only if the client sends no page_size 
  param
      page_size_query_param = "page_size"   # tells DRF to accept ?page_size=N 
  from the client
      max_page_size = 100         # server-side ceiling, regardless of what 
  client requests

  page_size_query_param is the key line — it tells DRF's PageNumberPagination to
  read page_size off the query string on every request and use that value
  instead of the default, as long as it's ≤ max_page_size. Without that line,
  page_size would in fact be fixed at 10 with no client override possible.

  So the actual flow:
  - Frontend's fetchJobs(page, pageSize) in api/jobs.ts sends
  ?page=1&page_size=5 (or whatever the "Rows per page" dropdown is set to)
  - Backend reads that param and slices the queryset to exactly that size
  - If the client omitted page_size entirely, the server falls back to 10
  - If the client asked for something absurd like page_size=100000, the server
  clamps it to 100

  This is confirmed by two of the backend tests:
  test_list_page_size_query_param_overrides_default (client's page_size=3 is
  honored) and test_list_page_size_is_capped_at_max (client's page_size=1000
  gets capped to 100). So "hardcoded" only applies to the default and the max —
  the actual per-request value is fully client-driven within that range.

❯ ok awesome. How are we tracking what has been retrieved on the backend? Is    
  there a blur token of some sort?                                              

⏺ No token of any kind — DRF's PageNumberPagination is completely stateless. The
  server tracks nothing about prior requests; every request independently
  recomputes its result from page and page_size alone.

  Concretely, in get_queryset() (backend/jobs/views.py), the queryset is just an
  ordered list of all jobs. When a request comes in with page=2&page_size=10,
  Django's paginator does the arithmetic itself:

  offset = (page - 1) * page_size   # = 10
  limit  = page_size                # = 10

  and translates that into SQL with LIMIT/OFFSET:

  SELECT ... FROM jobs_job ORDER BY created_at DESC LIMIT 10 OFFSET 10;

  Nothing is remembered between calls — if you request page 2 twice in a row,
  you get identical rows both times (assuming the data hasn't changed), purely
  because the same offset/limit math produces the same query.

  What you might be thinking of is cursor-based pagination (DRF has a
  CursorPagination class for this) — there, instead of a page number, the client
  passes back an opaque token encoding "the last row you saw" (e.g. its id or
  timestamp), and the server fetches rows strictly after that marker. That's the
  "token" pattern used by APIs like GitHub's or Stripe's.

  Why it matters here: offset/limit pagination has a real tradeoff — since each
  page is recomputed fresh from current data rather than a stable snapshot,
  results can drift if rows are added/removed between page loads. Example:
  you're viewing page 2; someone deletes a row from page 1; the next time you
  load page 2, everything shifts up by one and you silently see a row you
  already saw (or miss one). I already handled the specific case where your own
  delete shrinks the total page count out from under you (the client-side
  clamping logic from earlier), but general drift from other concurrent activity
  isn't addressed — cursor pagination is immune to that class of bug, offset
  pagination isn't. For this app's scale and single-user-at-a-time usage pattern
  it's a reasonable tradeoff, but worth knowing if this ever needs to handle
  concurrent editors.

⏺ How the debouncing works

  There are two separate pieces of state for the search box, which is the whole
  trick:

  searchInput      (raw, updates on every keystroke — instant, local only)
          ↓ useDebouncedValue(searchInput, 300)

  useDebouncedValue (frontend/src/utils/useDebouncedValue.ts):
  export function useDebouncedValue<T>(value: T, delayMs: number): T {
    const [debounced, setDebounced] = useState(value);

    useEffect(() => {
      const timeout = setTimeout(() => setDebounced(value), delayMs);
      return () => clearTimeout(timeout);
    }, [value, delayMs]);

    return debounced;
  }

  Every time searchInput changes (i.e., every keystroke), this effect schedules
  a timer to copy that value into debounced after 300ms. But — critically — the
  return () => clearTimeout(timeout) cleanup function runs before the effect
  re-fires on the next keystroke. So if you type "E", "T", "L" in quick
  succession:
  1. Type "E" → schedules "commit debounced='E' in 300ms"
  2. Type "T" (50ms later) → cancels that pending timer, schedules a new one for
  "debounced='ET' in 300ms"
  3. Type "L" (50ms later) → cancels again, schedules "debounced='ETL' in 300ms"
  4. You stop typing → 300ms of silence passes → the timer finally fires →
  debounced becomes "ETL"

  Only the last scheduled timer ever survives to fire. That's the entire
  debounce mechanism — no library, just setTimeout + cleanup.

  In JobsPage.tsx:
  - The <input> is bound to searchInput, so it updates instantly on every
  keystroke — the box itself never lags.
  - The actual data-fetching effect depends only on debouncedSearch, not
  searchInput:

  - So a network request only fires once debouncedSearch actually changes —
  i.e., once per pause in typing, not once per keystroke. That's what we
  verified earlier: typing "ETL" (3 keystrokes) produced exactly 1 API call.

  Every other action that refetches the list (create/update/delete/paginate)

❯ django dev server vs gunicorn. what is the difference and why does it matter? 

  Searched for 2 patterns (ctrl+o to expand)

⏺ Confirmed — gunicorn is sitting in requirements.txt but nothing ever actually
  runs it. The live command (docker-compose.yml's backend.command) is python 
  manage.py runserver 0.0.0.0:8000, Django's built-in dev server.

  The core difference

  runserver is single-threaded and single-process by design. Django's docs
  literally warn against using it in production. It:
  - Handles one request at a time by default — a second request has to wait for
  the first to finish, even if the first is just waiting on a slow DB query
  - Has no process management — if it crashes, nothing restarts it; if it hangs,
  nothing routes around it
  - Auto-reloads on every file save (great for dev, actively dangerous in prod —
  a code edit mid-request would kill in-flight requests)
  - Was never optimized for throughput, security hardening, or concurrent
  connections — it exists purely so you don't need a real server just to develop
  locally

  Gunicorn is a production WSGI server. It:
  - Runs multiple worker processes (you configure the count, commonly 2 × CPU 
  cores + 1), so it can genuinely serve multiple requests in parallel instead of
  queuing them behind each other
  - Has real process management — if a worker crashes or hangs, the master
  process kills and replaces it, so one bad request doesn't take down the whole
  app
  - Has no filesystem-watching/auto-reload overhead, and is built to just serve
  traffic efficiently
  - Is what Django's own docs, and basically every production deployment guide,
  recommend pairing with a reverse proxy (nginx) in front of it

  Why it matters here specifically

  Right now, our backend container can only truly process one HTTP request at a 
  time. Under the "high load and throughput" scenario you flagged for this next
  phase — say, several users hitting GET /api/jobs/ while someone else is
  mid-POST/PATCH — those requests queue up serially rather than running
  concurrently, even though the actual DB work each one does is small. This is
  the single highest-impact, lowest-risk fix on the list: gunicorn is already in
  requirements.txt, unused, so swapping the command: in docker-compose.yml (and
  adding a worker-count flag) gets real concurrency without touching a line of
  application code.

  One caveat worth flagging up front: gunicorn's auto-reload story is different
  from runserver's live volume-mount hot-reload we rely on for local dev — so if
  we make this switch, we'd want it to apply to a production-style config, not
  replace the dev experience we've built everything around.

⏺ Confirmed — CONN_MAX_AGE isn't set anywhere in settings.py, so Django's
  default (0) applies. That's actually relevant to your question, so let me
  untangle two things that are stacked on top of each other here.

  Your "one connection at a time" claim — partially right, for a different 
  reason than it sounds

  It's true that right now, at any given moment, there's effectively at most one
  active DB connection — but that's a side effect of runserver only handling 
  one request at a time, not a limit on connections themselves. Nothing about
  Postgres or Django caps you at one connection; it's that there's only ever one
  request in flight to need one.

  Separately — and this is the part CONN_MAX_AGE controls — Django's default 
  behavior, regardless of server, is to open a brand-new connection to Postgres
  for every single request, then close it when that request finishes. So even
  under runserver's one-at-a-time model, we're already paying "connect → auth
  handshake → query → disconnect" on every request, when we could be paying
  "connect → auth handshake" once and just "query" many times.

  What changes once we add gunicorn

  This is the important part: switching to gunicorn with, say, 4 worker
  processes doesn't just add concurrency — it also multiplies the connection 
  churn, because each worker is a separate OS process and (without pooling) each
  one independently opens+closes its own fresh Postgres connection per request
  it handles. So 4 workers under load = up to 4x the connect/disconnect overhead
  happening simultaneously, all hitting Postgres at once.

  What pooling actually buys us

  Two concrete things:

  1. Avoids repeating expensive setup on every request. A new Postgres
  connection isn't cheap — it's a TCP handshake, an authentication round-trip,
  and (this is Postgres-specific and easy to miss) Postgres forks an entire new 
  OS process per connection. That process fork has real memory and CPU cost.
  Doing that fresh on every single API request, when the actual query might take
  a few milliseconds, means the connection overhead can dwarf the actual work
  being done.
  2. Protects Postgres's connection budget as concurrency goes up. Postgres has
  a hard max_connections ceiling (commonly defaults to 100), and each open
  connection holds real memory on the Postgres side for its lifetime. Once
  gunicorn lets us run many workers concurrently, we risk many simultaneous open
  connections competing for that fixed budget — pooling caps how many actual
  Postgres connections exist regardless of how much app-level concurrency we
  throw at it.

  There are two tiers of fixing this, worth treating as separate steps:

  - CONN_MAX_AGE (a Django setting, zero new infra): tells Django to keep a
  connection alive and reuse it across multiple requests within the same worker 
  process, instead of reconnecting every time. Cheap, simple, no new moving
  parts — but each worker still has its own separate connection, so it doesn't
  help if we scale worker count way up.
  - PgBouncer (an external pooler sitting between the app and Postgres): lets
  all gunicorn workers share a much smaller pool of real Postgres connections.
  This is what actually decouples "how many app processes we run" from "how many
  Postgres connections exist" — the real fix once we're talking about genuine
  high concurrency.

  Given where we are, CONN_MAX_AGE is the natural first step alongside the
  gunicorn switch (near-zero cost), with PgBouncer as the follow-up once/if
  we're actually running enough concurrent workers for it to matter.

⏺ Right, and that's the key variable — cache value scales with the read-to-write
  ratio for the same data, not just "does the data change." A few things worth
  adding to that:

  Cache granularity matters as much as write frequency. If we cache the list
  response (page of jobs with embedded statuses), any single job updating
  anywhere invalidates that whole cached page — since the list aggregates across
  many entities, it's maximally sensitive to writes no matter how localized
  they are. But if we instead cache per-job responses (GET /api/jobs/<id>/),
  only that one job's entry goes stale when it updates; the other 49 jobs on the
  page stay cache-valid. So the real question isn't "how often do jobs change"
  in aggregate, it's "how often does this specific job change, and are writes
  concentrated on a small subset of jobs (actively running ones) while most sit
  in terminal states for a long time." If the latter, per-job caching could
  still pay off even with plenty of write activity happening somewhere in the
  system — list-level caching is what falls apart fastest.

  There's also a middle ground worth separating out from "should we cache at 
  all": micro-caching / request coalescing. Instead of caching to survive across
  write cycles, cache for a tiny TTL (e.g. 1–2s) purely to deduplicate a burst
  of identical concurrent reads — say 50 clients polling the same page in the
  same second. That's a different justification than "reduce long-term DB load,"
  and it doesn't fight your invalidation concern at all, since it's designed to
  expire faster than any reasonable write cadence anyway. Worth keeping on the
  table as a narrow, low-risk win independent of the bigger "should we cache the
  list" question, which I'd say stays genuinely TBD pending real traffic data.

  On ETags — I'd actually split this into two mechanisms that do different jobs,
  since "ensure clients have the latest version" describes only one of them:

  - Conditional GET (If-None-Match → 304) is a bandwidth/compute optimization,
  not a correctness guarantee. It just means "don't re-send/re-serialize data
  the client already has." A client that never re-validates can still act on
  stale data it's holding — ETags don't push updates to anyone.
  - Conditional write (If-Match on PATCH → 412 if stale) is the actual
  correctness mechanism — reject a write if it's based on a version the client
  no longer has.

  Here's a subtlety specific to our domain: our PATCH is append-only (it adds a
  new JobStatus row, never overwrites), so we don't have a classic lost update
  bug — if two people both PATCH concurrently, both transitions get recorded in
  history, nothing's silently discarded. What we do have is a stale-decision 
  problem: User B loads the page seeing RUNNING, User A already moved it to
  SUCCEEDED a moment earlier, and User B — still looking at stale RUNNING —
